## Context

本プロジェクトは、GitHub Pages を利用してサーバーレスでホストされる、完全無料の英語学習用Webアプリケーション（英語ラジオ・ニュース学習アプリ）です。
静的ホスティングであるため、バックエンドでの動的処理（スクレイピング、翻訳API呼び出し等）をそのままクライアントサイドで行うと、CORS制限やAPIキーの露出問題が生じます。これらを解消しつつ、無料で持続可能なシステムデザインを設計します。

## Goals / Non-Goals

**Goals:**
- GitHub Pages 上で完全無料で動作するSPA（Single Page Application）の構築。
- 日本の主要ニュースRSSから定期的に最新のニュースを取得し、Gemini API（無料枠）で「英語サマリー」「学習用英文」「対訳日本語文」を生成し、静的配信する仕組みの構築。
- Web Speech APIを利用した、英文と和文が同期してハイライト表示される音声再生プレイヤーUIの提供。

**Non-Goals:**
- 有料のTTS（Text-to-Speech）や翻訳APIの導入（運用の完全無料化を最優先するため）。
- データベースや常時稼働サーバーの構築。
- ニュースのオリジナル画像やマルチメディアコンテンツの完全な同期（テキストと音声のみに焦点を絞る）。
- 任意のテキストや外部URLをリアルタイムで要約・翻訳する動的カスタム機能（配信されたニュース教材のリスニング学習に特化し、完全静的な構成にするため）。

## Decisions

### 1. データ配信アーキテクチャ: GitHub Actions による静的データジェネレーション
- **意思決定**: 1日に複数回、GitHub Actionsのワークフローを起動し、Pythonスクリプトで日本の主要ニュースRSSを取得。リポジトリのSecretsに保存した管理者の Gemini API キーを使用して翻訳・要約処理を行い、結果を `data/news.json` という静的JSONファイルとしてコミット・プッシュする。
- **ニュース取得手法の詳細**:
  - **取得元RSS**: NHKニュースのRSSフィードである `https://news.web.nhk/n-data/conf/na/rss/cat0.xml` を利用します。
  - **記事本文の取得・生成方式**: 実際のニュース詳細に基づいた質の高い学習コンテンツを作成するため、RSSから取得した各ニュース記事のリンク（URL）先へアクセスし、本文テキストをフェッチします。具体的には、Pythonスクリプトでニュース解析専用ライブラリである **`newspaper3k`** を使用し、各URLから本文テキスト（およびタイトルやメタデータ）を自動的にパース・抽出します。この抽出したニュースの**「本文全文」**を Gemini 1.5 Flash に入力し、Gemini APIに対して「このニュース本文の内容に基づき、英語リスニング学習に適した5〜10文程度の自然な英文ニュース記事、および文単位で対応する日本語対訳、英語の要約を生成してください」と指示することで、ニュースの全内容を正確に反映した英語学習テキストを生成します。
- **理由**: ブラウザからRSSフィードを直接取得する際のCORS制限を回避できます。また、管理者のGemini APIキーをフロントエンドに露出させることなく、バックエンド（CI/CD環境）で安全に利用して自動生成データを作成できます。ユーザーは静的JSONをロードするだけなので、超高速かつAPIリクエスト制限を気にせず利用できます。
- **代替案**: クライアントサイドCORSプロキシ、またはRSSの概要（一文程度）のみに基づくAI生成。しかし、プロキシはサービス停止のリスクが高く、概要のみではニュースの詳しい内容を反映した英語の長文を作れないため、RSS詳細リンクからの本文フェッチ方式を採用しました。個別記事のスクレイピングに `newspaper3k` ライブラリを採用したことで、特定のニュースサイトのHTML構造変更に依存しない堅牢なテキスト抽出が可能になります。

### 2. 音声読み上げ技術: Web Speech API (SpeechSynthesis)
- **意思決定**: すべてのモダンブラウザに標準搭載されている `SpeechSynthesis` API を使用して、デバイス内で直接英語テキストの音声合成を行う。
- **理由**: 完全無料かつ制限なしで利用可能であり、追加のライブラリやAPIキーも不要です。再生速度（rate）の制御、およびデバイスが持つ英語話者の選択機能が標準で提供されています。
- **代替案**: Google Cloud Text-to-Speech や Amazon Polly などのクラウドTTS。非常に高品質だが、無料枠を超えると課金が発生するため却下。

### 3. 対訳同期ハイライトの制御方法: 文（Sentence）分割再生方式
- **意思決定**: 英文と和文は、Gemini APIから「英文と和文が対になったオブジェクトの配列（例: `[{en: "...", ja: "..."}]`）」としてJSONで取得する。プレイヤーでの再生時は、文章全体を一括で読み上げるのではなく、1文（またはパラグラフ）ごとに `SpeechSynthesisUtterance` オブジェクトをキューに投入して順番に再生する。
- **理由**: Web Speech APIの境界検知（`onboundary`）イベントは、ブラウザやOS、選択した音声によって文字インデックスの挙動が異なり、非常に不安定です。一方、文ごとに発話を切り分けてキューイングし、各発話の `onstart` イベントで画面上の対応する英文要素をハイライトし、和文を表示する方式にすることで、ブラウザ環境に依存しない確実な同期表示を実現できます。

## Risks / Trade-offs

- **[Risk]** Web Speech APIの音声品質は、ユーザーのOSやブラウザ環境（iOS、Android、Windows、macOS）に強く依存する。
  - **[Mitigation]** アプリ初期化時にブラウザで利用可能な英語の音声一覧を取得し、ユーザーが任意の話者（Google US Englishなど）を選択できるUIを提供する。また、音声の違いによってレイアウトが崩れないようなレスポンシブデザインを採用する。
- **[Risk]** RSS配信元のフォーマット変更や、Gemini APIの出力フォーマット揺れによるパースエラー。
  - **[Mitigation]** Pythonビルドスクリプトとフロントエンドに厳密なスキーマ検証を導入する。また、Gemini APIへのシステムプロンプトでJSONフォーマットでの出力を徹底（`response_mime_type: "application/json"`）し、パース失敗時はフォールバックデータ（既存のニュースデータ）を使用することで、アプリが真っ白になるのを防ぐ。
- **[Risk]** GitHub Actionsの実行時間制限（無料枠）。
  - **[Mitigation]** リポジトリをパブリックに設定することで、GitHub Actionsは完全に無料となります。また、処理自体は軽量なPythonスクリプトのみであるため、数分以内にビルド・デプロイが完了します。
